## 2. Devpost Text Description (copy-paste)

**FirstResponder — Crisis & incident coordination agent for Slack**

When an emergency hits — an active threat at a school, a fire at a community
center, a ransomware attack on a nonprofit — the organizations that need crisis
coordination tools the most are the ones that can't afford them. PagerDuty and
Opsgenie cost $9–21 per user per month; a 50-person school pays $12K/year. So
they fall back on panic in group chats, phone trees that fail, and paper rosters
nobody can find. Personnel accountability that should take 3 minutes takes 30.

FirstResponder gives every Slack workspace a full crisis coordination system for
free. One command turns your workspace into a coordination center in under 60
seconds.

**What it does**
- **`/crisis start <type> <description>`** declares an incident with a unique ID
  and severity, posts a type-specific response playbook, and auto-seeds a
  personnel-accountability roster from the channel's members.
- **Live personnel accountability** — people check in by reacting with an emoji
  (✅ safe, 🆘 need-help, 🚑 injured, 🚪 evacuated) or `/crisis checkin`. The agent
  tracks who is safe and, crucially, **who is still missing**.
- **A building-aware AI agent** (`@FirstResponder`) with **32 tools** across three
  intelligence layers. It doesn't give generic advice — it knows *this* building:
  "Mrs. Davis is in Room 104, floor 1, uses a wheelchair — elevator required.
  Nearest AED is in Hallway B. East stairwell is blocked by the fire; route via
  Door 7."
- **Situation reports** generated by reading the actual incident-channel
  conversation through the **Slack MCP Server** (Real-Time Search API).
- **A learning engine** — every incident persists to SQLite. Lessons learned in
  incident N automatically surface during incident N+1: "Based on 3 past fire
  incidents, average resolution is 45 minutes. Last time we learned: shut gas
  first." After-action reports compare resolution time to the historical average.
- **Scenario-specific intelligence** — fire triggers gas-shutoff + hazmat +
  evacuation routing; a data breach triggers PII inventory + regulatory
  notification windows (GDPR 72h, HIPAA 60d) + legal on-call; an outage triggers
  dependency blast-radius + runbook + vendor SLA. Ten crisis types, each with its
  own protocol.

**How it's built**
Bolt for Python (Socket Mode) handles Slack events; the Claude Agent SDK
orchestrates the 32-tool agent with session memory and streaming responses; a
local SDK-MCP server exposes the tools and the remote Slack MCP Server provides
channel search. Two SQLite databases — a learning store and a knowledge base —
give it memory and building awareness with zero external infrastructure. Load
your building via drag-and-drop CSVs; no code required.

## 3. Impact Statement (copy-paste — required for Agent for Good)

FirstResponder targets a real equity gap in public safety: professional incident
management is priced for enterprises, so schools, nonprofits, houses of worship,
and small community organizations — exactly the places where a coordinated
emergency response saves lives — go without.

- **Cost:** $0, versus $9–45/user/month for PagerDuty/Opsgenie/ServiceNow. A
  50-person school saves ~$12,600/year.
- **Speed:** From "emergency reported" to "full coordination active" in under 60
  seconds — playbook posted, roster seeded, accountability tracking live.
- **Accessibility built in:** The knowledge base tracks mobility limitations,
  medical notes, and floor wardens, so the agent prioritizes people who need
  assistance during an evacuation and routes around blocked exits.
- **Who it serves:** K-12 schools (active-shooter/fire/severe-weather drills),
  nonprofits facing the same cyber threats as enterprises with none of the
  budget, houses of worship and community centers run by volunteers with no IT
  staff, and open-source projects coordinating security/outage response.
- **Gets smarter for everyone:** Because it learns from every incident, an
  organization's response maturity compounds over time instead of resetting with
  staff turnover.

It runs where these teams already are — Slack — with no servers to manage and no
per-seat cost. FirstResponder is not a replacement for calling 911; it coordinates
organizational response alongside professional emergency services.

## 7. Known Limitations (be ready if judges probe)

These are deliberately deferred, not bugs to hide.

1. ~~In-memory crisis state is not rehydrated from SQLite on restart.~~ **FIXED.**
   The manager now reloads all non-resolved incidents (with check-ins, roster,
   timeline, and SITREPs) on startup, and resumes its incident-ID counter so
   fresh IDs can't collide with pre-restart ones. An app restart mid-incident no
   longer orphans the incident. Verified across a simulated restart.
2. **Synchronous SQLite inside async handlers.** Fine for single-user/scripted
   use; under real concurrency it can briefly block the event loop. Would move to
   `asyncio.to_thread` / an async driver for production scale.
3. **After-action historical comparison includes the just-resolved incident** in
   its own average (the incident is persisted as `resolved` before the report is
   generated), slightly biasing the "faster/slower than average" line.
4. **Roster is seeded from channel members**, not the knowledge-base personnel
   directory (the demo directory uses placeholder Slack IDs). This is the correct
   choice for real accountability, but worth knowing for the demo narrative.
